## 5. 配置重定向URL ⭐（最关键步骤）

### 为什么需要配置重定向URL？

当用户在飞书授权页面点击"同意"后，飞书会自动将用户**重定向回**你指定的URL，并携带授权码（code）。这个URL必须**提前在飞书平台注册**，否则会被拒绝！

### OAuth回调流程图

```
用户浏览器                    飞书服务器                   你的服务器
    │                           │                           │
    │  1. 点击"飞书登录"         │                           │
    ├─────────────────────────► │                           │
    │                           │  2. 显示授权页面            │
    │◄──────────────────────────┤                           │
    │                           │                           │
    │  3. 用户同意授权           │                           │
    ├─────────────────────────► │                           │
    │                           │  4. 重定向到回调URL        │
    │                           │     (携带code参数)        │
    │◄──────────────────────────────────────────────────────┤
    │                                                      │
    │  5. GET /api/auth/callback?code=xxx                  │
    ├─────────────────────────────────────────────────────►│
    │                                                      │
    │                      6. 返回登录成功页面               │
    │◄──────────────────────────────────────────────────────┤
```

### Step 1: 找到安全设置
在应用详情页左侧菜单中，点击：

```
安全设置 → 重定向 URL
```

或者路径：
```
安全设置 → 重定向 URL → 添加重定向 URL
```

### Step 2: 添加开发环境URL（必填）

点击 **"+ 添加重定向 URL"** 按钮，输入：

```
http://localhost:3000/api/auth/callback
```

**详细说明**:
- `http://localhost:3000`: 你本地开发服务器的地址和端口
- `/api/auth/callback`: 后端处理OAuth回调的路由路径

⚠️ **注意事项**:
- ✅ 必须包含完整路径（协议+域名+端口+路径）
- ✅ 不能有多余的空格或特殊字符
- ✅ 不能以 `/` 结尾（除非API确实需要）
- ❌ 不要写成 `http://localhost:3000` (缺少路径)
- ❌ 不要写成 `localhost:3000/api/auth/callback` (缺少协议)

### Step 3: 添加生产环境URL（如已确定）

如果你的生产环境域名已确定，继续添加：

```
https://qa-platform-1-zat2.onrender.com/
```

将 `yourdomain.com` 替换为你的实际域名。

**示例**:
```
https://qaui.example.com/api/auth/callback
https://app.yourcompany.com/api/auth/callback
```

### Step 4: 保存配置
1. 点击 **"确定"** 或 **"保存"** 按钮
2. 系统可能会要求输入密码或进行二次验证
3. 保存成功后会显示在列表中

### 最终效果预览

配置完成后，你应该看到类似这样的列表：

```
✅ http://localhost:3000/api/auth/callback      [开发环境]
✅ https://qaui.example.com/api/auth/callback   [生产环境]（可选）
```

## 6. 申请权限

### 6.1 需要申请的权限列表

对于QAUI系统的飞书登录+服务台跳转功能，你需要申请以下权限：

#### **基础权限（登录功能必需）**

| 权限名称 | 权限标识符 | 必要性 | 说明 |
|----------|------------|--------|------|
| **获取用户基本信息** | `authen:user_id:readonly` | ✅ 必需 | 获取用户的姓名、头像等基础信息 |
| **获取用户邮箱** | `authen:user_email:readonly` | ⚠️ 可选 | 获取用户邮箱（用于显示） |
| **获取用户手机号** | `authen:user_mobile:readonly` | ⚠️ 可选 | 获取用户手机号（用于显示） |

#### **增强权限（智能服务台跳转 - 消息卡片功能）**

> ⭐ **新增于 v2.0** - 如果需要使用"智能跳转服务台"功能（发送消息卡片到服务台），请额外申请以下权限：

| 权限名称 | 权限标识符 | 必要性 | 说明 | 功能场景 |
|----------|------------|--------|------|----------|
| **以机器人身份发送消息** | `im:message:send_as_bot` | ✅ 增强功能必需 | 允许应用向群聊/会话发送消息 | 向MTDS服务台发送用户咨询信息 |
| **获取与发送单聊/群组消息** | `im:message` | ✅ 增强功能必需 | 读取和发送消息的通用权限 | 发送结构化消息卡片 |
| **获取群组信息** | `im:chat:readonly` | ⚠️ 推荐 | 读取群组基本信息（可选） | 验证服务台会话是否存在 |

**权限对比表**:

| 功能模块 | 所需权限 | 不申请的影响 |
|----------|----------|--------------|
| **基础登录** | 仅需要基础权限（3个） | 无法登录 |
| **普通服务台跳转** | 仅需要基础权限（3个） | 可以跳转，但无法发送消息卡片 |
| **智能服务台跳转** | 基础权限 + 增强权限（共5-6个） | 降级为普通跳转模式 |

**推荐策略**: 
- ✅ **一次性申请所有权限**（推荐）- 避免多次审批流程
- 🔲 **分阶段申请** - 先申请基础权限上线登录功能，后续再申请增强权限

### Step 1: 进入权限管理页面
在应用详情页左侧菜单中，点击：

```
权限管理 → 搜索并开通权限
```

或者直接访问：
```
https://open.feishu.cn/app/cli_a97ad490803d5bd3/permission
```

### Step 2: 搜索并添加权限

#### 方法A: 通过搜索框查找
1. 在搜索框中输入：`authen:user_id:readonly`
2. 点击搜索结果中的 **"申请权限"** 或 **"+"** 按钮

## 9. 常见问题排查

### Q1: 报错 "redirect_uri_mismatch"

**现象**：点击登录后，飞书页面显示此错误

**原因**：重定向URL配置不正确或不完整

**解决方案**：
1. 检查飞书平台配置的重定向URL是否为：
   ```
   http://localhost:3000/api/auth/callback
   ```
2. 确保没有多余的空格、大小写错误、缺少协议头等
3. 确保端口正确（默认是3000）
4. 修改后需要**重新保存**配置

**快速诊断命令**：
```bash
# 在浏览器控制台中执行
console.log(window.location.origin + '/api/auth/callback');
# 输出应该是: http://localhost:3000/api/auth/callback
```

---

### Q2: 报错 "invalid_grant" 或 "code expired"

**现象**：回调接口报错，提示授权码无效或已过期

**原因**：
- 授权码（code）只能使用**一次**
- 授权码有效期很短（通常几分钟内过期）
- 可能在测试时重复使用了旧的URL

**解决方案**：
1. 重新发起登录流程（不要刷新回调URL）
2. 确保回调逻辑中没有重复请求
3. 检查网络延迟导致的问题

---

### Q3: 报错 "missing app id or app secret"

**现象**：回调接口返回此错误

**原因**：
- 后端配置文件中的 App ID 或 App Secret 不正确
- 或者凭证被修改但后端未更新

**解决方案**：
1. 打开 [backend/config/feishu.js](../backend/config/feishu.js)
2. 对比飞书平台的凭证信息
3. 确保完全一致（包括大小写）
4. 修改后重启后端服务

**验证方法**：
```javascript
// 在 config/feishu.js 中临时添加日志
console.log('App ID:', appId);
console.log('App Secret:', appSecret);
// 启动服务后查看输出是否正确
```

---

### Q4: 报错 "permission denied" 或 "scope not granted"

**现象**：获取用户信息时返回权限不足错误

**原因**：
- 未申请必要的权限
- 权限申请尚未审批通过
- 权限标识符写错

**解决方案**：
1. 检查是否已申请 `authen:user_id:readonly` 权限
2. 确认权限状态为 **"已开通"**
3. 如果是刚申请的，等待审批通过（通常几分钟到几小时）

---

### Q5: Session丢失（刷新页面后需要重新登录）

**现象**：登录成功后，刷新页面又变成未登录状态

**原因**：
- Cookie未正确设置
- 浏览器禁用了Cookie
- Session中间件配置问题

**解决方案**：
1. 检查浏览器是否允许Cookie（特别是localhost）
2. 确认 [backend/app.js](../backend/app.js) 中session配置：
   ```javascript
   cookie: {
     secure: false,  // 开发环境必须为false
     httpOnly: true,
     maxAge: 24 * 60 * 60 * 1000
   }
   ```
3. 清除浏览器缓存和Cookie后重试

---

### Q6: 生产环境HTTPS问题

**现象**：生产环境下无法登录或Cookie无效

**原因**：生产环境设置了 `cookie.secure = true` 但未启用HTTPS

**解决方案**：
- **方案A（推荐）**: 安装SSL证书启用HTTPS
  ```bash
  # Let's Encrypt 免费证书
  certbot --nginx -d yourdomain.com
  ```
- **方案B（临时）**: 将secure设为false（不推荐生产环境长期使用）

---

### Q7: CORS跨域问题

**现象**：前端调用API时报错 "Access-Control-Allow-Origin"

**原因**：前后端不在同一域名下，或协议不同（http vs file://）

**解决方案**：
1. 确保通过HTTP协议访问（不是file://）
2. 开发环境使用：`http://localhost:3000/test-login.html`
3. 生产环境确保前后端同域或配置CORS白名单
